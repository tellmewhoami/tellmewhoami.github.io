DailyStudy20260224-Template Method

Template Method 模式(对应hibim代码中的CToolCmd命令基类)

一、Template Method（模板方法）模式概述

Template Method 是一种行为型设计模式，它在基类中定义一个操作中的算法骨架，而将一些步骤延迟到子类中实现。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法中的某些特定步骤。

模式核心思想

  • 固定流程：算法的整体结构由基类中的模板方法决定，顺序固定。
  • 可变细节：部分步骤由子类实现，允许不同的行为。
  • 钩子操作（Hook）：基类可以提供默认实现或空实现，子类可选择性地覆盖，以影响算法的某些部分。加上virtual修饰符；

三、模板方法模式的优缺点

优点

  1. 代码复用：将公共行为提取到基类，避免子类重复代码。
  2. 扩展灵活：子类只需实现特定步骤，即可产生新的行为。
  3. 遵循开闭原则：算法骨架封闭，具体步骤开放扩展。
  4. 控制反转：父类调用子类方法（好莱坞原则），子类不决定流程。

缺点

  1. 继承限制：子类必须继承基类，Java/C# 单继承可能限制灵活性。
  2. 维护难度：基类中步骤越多，子类依赖越强，修改骨架可能影响所有子类。
  3. 可能违反里氏替换原则：若子类重写钩子方法改变预期行为，可能破坏父类设计意图。
  4. 调试复杂：流程分散在多个类中，跟踪执行顺序需要跨类查看。

四、回答用户问题

1. 请问这是一种设计模式吗？

是的，Template Method 是 GoF 经典设计模式之一，属于行为型模式。它通过继承实现算法结构的复用，是面向对象设计中非常基础且常用的模式。你的代码 DoCmd() 方法正是该模式的典型实现。

2. 对于模板方法，还有什么其他类似的设计模式？

以下模式与 Template Method 在目的或结构上有相似之处，但各有侧重：

模式 相似点 主要区别
策略模式 都允许算法变化 模板方法使用继承改变算法部分步骤；策略模式使用组合，将整个算法封装为独立对象，可动态替换。
工厂方法模式 父类定义创建对象的接口，子类决定实例化哪个类 工厂方法专注于对象创建，模板方法关注算法步骤。
建造者模式 分步骤构建复杂对象 建造者模式将构建过程与表示分离，通过 Director 控制步骤顺序；模板方法的步骤顺序固定于父类。
钩子方法 都是通过继承提供扩展点 钩子方法不是一种完整模式，而是模板方法中常用的技术手段。
回调/委托 将部分行为委托给外部 回调通常通过接口或委托实现，更灵活；模板方法依赖继承，静态绑定。

简要对比示例

  • 策略模式：若将 DoUI() 和 Analyse() 设计为可动态替换的策略接口，则可以在运行时更换交互方式和业务逻辑，而无需定义新的子类。
  • 工厂方法：如果 Reset() 需要创建某个对象，可将其抽象为工厂方法，让子类决定创建哪种具体对象。

五、总结

Template Method 模式通过继承优雅地实现了“不变封装变”的设计原则，是框架设计的基石之一（如 .NET 中的 Stream 类、Asp.Net 生命周期等）。理解它能帮助你写出更易扩展和复用的代码。

Revit开发–当前视图不是可图形化视图错误

这个报错的意思是：当前激活的视图不是“可图形化”的视图（non-graphical），因此 Revit 不允许在该视图里执行任何需要“拾取/点选”的操作（pick operations），例如：

  • UIDocument.Selection.PickObject(...)
  • PickObjects(...)
  • PickPoint(...)
  • 以及一些需要把焦点切到图形窗口来交互选择的 API

常见触发场景：当前激活的是 项目浏览器、明细表(Schedule)、报表、系统/浏览器面板、或者其它非图形窗口；或者处在某些模式下导致视图无法接收拾取焦点。

解决思路

确保激活的窗口是图形视图，不要在非交互上下文里调用 Pick；
