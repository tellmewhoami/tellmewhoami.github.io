一般来说，饿汉式就是在类加载时就立即创建类的实例，即在类的字段中直接创建对象的实例，然后通过一个public的属性暴露对象；他的特点时线程安全，但是在类加载的时候就会创建实例，所以一定会占用大量内存。
懒汉式则不在类加载的时候就创建实例，而是在调用公共属性的get方法的时候才创建实例，不同的实现方式效果不同，一般建议在get方法创建实例外再讨一个锁结构，确保线程安全，在锁外部加一个if判断确保在创建实例后不需要反复加锁。同时也可以使用C#的Lazy版，是C#标准库中定义的一个泛型类，核心作用是提供一种线程安全的机制，将对象的创建推迟道首次访问对象的value属性时才进行。
手动加锁机制本质也是懒汉式的一种，通过lock关键字确保在多线程环境下共享资源的同步访问，当一个线程在访问时，其他线程会在锁结构外排队等待。

饿汉式单例

基本概念

饿汉式（Eager Initialization）在类加载时就立即创建实例，不管是否会被使用。

特点分析

特性 说明
线程安全 ✅ 天生线程安全（CLR保证静态初始化只执行一次）
延迟加载 ❌ 不支持（类加载即创建）
性能 访问快（无需同步），但启动慢
资源占用 始终占用内存，即使从未使用
异常处理 构造函数异常会导致整个类型不可用

适用场景

  • 实例一定会被使用
  • 创建开销小，不影响启动
  • 对性能要求极高（无锁访问）

懒汉式单例

基本概念

懒汉式（Lazy Initialization）在第一次被访问时才创建实例。

不同实现方式

1. 基础版（非线程安全）

2. 双重检查锁定版（线程安全）

3. 使用 Lazy<T> 版（推荐）

特点分析

特性 说明
线程安全 ⚠️ 取决于实现（双重检查或Lazy保证）
延迟加载 ✅ 支持（首次访问才创建）
性能 首次访问有开销，后续无锁
资源占用 按需加载，节省内存
实现复杂度 较高（需考虑线程安全）

适用场景

  • 实例可能不会被使用
  • 创建开销大（如加载文件、建立连接）
  • 启动时间敏感的应用

锁对象的深入理解

问题 为什么这样设计
为什么用 object 类型？ lock 要求引用类型，object 最简单轻量
为什么用 private？ 防止外部代码意外锁定同一对象导致死锁
为什么用 readonly？ 确保锁对象不会被重新赋值
为什么专用对象？ 避免意外锁定其他用途的对象

注意事项

  1. 锁粒度：尽量缩小锁的范围，只保护必要的代码；
  2. 死锁预防：总是按相同顺序获取多个锁
  3. 异常安全：lock 语句会自动释放锁（即使异常）
  4. 性能考虑：锁有开销，避免过度使用

三种方式对比

维度 饿汉式 懒汉式（双重检查） 懒汉式（Lazy<T>） 手动加锁
线程安全性 安全 安全 安全 ⚠️ 手动保证
延迟加载 ❌ ✅ ✅ 不适用
代码复杂度 低 高 低 中
性能 高（无锁） 中 高 中
资源占用 始终占用 按需占用 按需占用 不适用
适用场景 必须使用的资源 可能不用的资源 可能不用的资源 任何需同步的场景

Revit开发中的应用建议

场景1：日志系统 - 推荐懒汉式 + 静态锁

场景2：Revit元素缓存 - 推荐饿汉式

场景3：复杂资源加载 - 推荐Lazy<T>

选择指南

总结要点

  1. 饿汉式：最简单、线程安全，但无延迟加载
  2. 懒汉式：支持延迟加载，需注意线程安全
  3. 手动加锁：lock 关键字保护共享资源
  4. 双重检查锁定：懒汉式 + 线程安全的优化方案
  5. Lazy<T>：现代C#推荐的懒加载实现

在Revit二次开发中，推荐：

  • 日志、配置：使用懒汉式 + 静态锁
  • Revit内置数据：使用饿汉式
  • 复杂资源：使用 Lazy<T>
  • 频繁写入：使用静态锁保护
